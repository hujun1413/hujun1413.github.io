Emulation of Software Faults: A Field Data Study and a Practical Approach（TSE06）（G-SWFIT）

介绍了实现无缺陷软件的困难，所以软件中有很多的Bug。

to improve the representativeness of injected
faults

  1. FIELD STUDY AND SOFTWARE FAULTS
    CLASSIFICATION
    扩展ODC将668个错误进行分类，分为assignment,checking,interface,algorithm,function,不同软件的以上的细分，再加missing,wrong,extraneous，找出每类错误最具代表性的是5种中的哪一个。
  2. EMULATION OF SOFTWARE FAULTS
    对有和无这些代表性错误的程序进行编译，找出有无错误造成的底层代码影响，找出正确和错误的区别，然后在可执行程序级别注入错误，验证了和在高层次的代码级别注入的效果是大体一致的。
    做一些study然后利用得到的结论找到一个切入点，只研究一个小点

Automatic Fault Injection for Driver Robustness Testing(ISSTA15)(ADFI)

提出自动生成错误注入用例的方法，叫BOUNDED TRACE-BASED ITERATIVE
FAULT GENERATION来防止出现测试用例爆炸增长：有相同调用栈的function错误注入不超过A,错误注入的函数不超过B
解决相同测试用例造成的调用栈不同
手动

LFI: A Practical and General Library-Level Fault Injector（DSN09）

介绍了一个函数库的错误注入器
先将函数库（如LibC库）反汇编，然后找到导出的函数API的返回值和一些side effect，生成中间类似XML语言的Profile，基于这个profile生成一个注入错误的函数库，执行应用程序调用这个函数库再检测错误。
优点：不需要函数库源码，全自动简单好用，提高代码覆盖率，能找到文档中可能漏的函数返回值，profile时间在分钟级，准确率80%—90%，overhead小。

An Extensible Technique for High-Precision Testing of Recovery Code(ATC10)

在LFI的基础上，增加精确错误注入的功能：开发了一种错误注入语言（接口和库），用于编写更复杂的错误注入测试用例：何时何处怎样注入错误
A fault injection trigger is a way to specify which fault
(“what”) to inject at which point in a program’s execution
(“when”) and in which call to the target library
(“where”).

Efficient Testing of Recovery Code Using Fault Injection(ACM11)
